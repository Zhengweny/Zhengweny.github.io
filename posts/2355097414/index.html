企业级代码部署级回滚

发表于2025-07-15|更新于2025-07-15|学习类CICD
|总字数:1.9k|阅读时长:8分钟|浏览量:

代码部署方式：

1）灰度部署（金丝雀）：生产环境要发布新代码时，不改变原有的版本v1，同时部署一个新版本v2应用作为”金丝雀”【在同一台服务器上，区别于蓝绿】，通过负载均衡的方式，让一部分用户测试新代码是否有问题，如果有问题，则撤销负载均衡，让用户访问原先v1版本的代码

2）蓝绿部署：就相当于有两套正式环境，当版本升级时，在一套环境中运行没问题，则切到对应的那套环境上（这种对于服务器成本比较高，比较少用）

3）A/B测试：就是公司分成了A组和B组团队，投入一样的多的资源，开发一样的东西，客户都会用，也就是一个团队开发得好就先用哪个团队的，但是两个版本都可供客户选择，都是生产环境（也很烧钱）

1、新建任务

2、自定义参数

设置3个参数选项：

  • METHOD=$1 对应 deploy、rollback
  • BRANCH=$2 对应 main、develop
  • GROUP=$3 对应 Group1（192.168.131.67：灰度环境的设备）、Group2（192.168.131.68/70：表示生产环境的设备）

3、构建环境（Shell脚本）

shel脚本存放位置：

#!/bin/bash

# 用法: bash deploy_gray.sh <deploy|rollback|offline|online> <main|develop> <Group1|Group2>

# 说明：所有远程操作均以 root 用户执行，如需更换为其他用户，请统一修改 ssh/scp 命令中的用户名部分。

export HOME=/var/lib/jenkins

chmod 600 /var/lib/jenkins/.ssh/id_rsa

chmod 700 /var/lib/jenkins/.ssh

set -e

# 参数校验

if [ $# -ne 3 ]; then

echo "用法: $0 <deploy|rollback|offline|online> <main|develop> <Group1|Group2>"

exit 1

fi

METHOD=$1

BRANCH=$2

GROUP=$3

# 配置区

REPO_URL="git@192.168.131.69:xingyun-system/devops.git"

WORKDIR="/tmp/gray_deploy"

DATE_TAG=$(date +%Y%m%d_%H%M%S)

PACKAGE_NAME="static_${DATE_TAG}.tar.gz"

REMOTE_BASE="/data/tomcat"

REMOTE_PACKAGES="$REMOTE_BASE/packages"

REMOTE_RELEASES="$REMOTE_BASE/releases"

REMOTE_LINK="/usr/local/tomcat/webapps/ROOT"

HAPROXY_HOST="192.168.131.66" # 你的HAProxy主机（VIP1）

HAPROXY_SOCK="/var/lib/haproxy/stats"

BACKEND_NAME_GROUP1="group1_backend"

BACKEND_NAME_GROUP2="group2_backend"

# 目标服务器

if [ "$GROUP" == "Group1" ]; then

SERVERS=("192.168.131.67")

BACKEND_NAME=$BACKEND_NAME_GROUP1

SERVER_NAMES=("tomcat1")

elif [ "$GROUP" == "Group2" ]; then

SERVERS=("192.168.131.68" "192.168.131.70")

BACKEND_NAME=$BACKEND_NAME_GROUP2

SERVER_NAMES=("tomcat2" "tomcat3")

else

echo "Group参数错误，只能为 Group1 或 Group2"

exit 2

fi

# 拉取代码并打包（保留 devops 目录名）

function git_and_package() {

GIT_SSH_COMMAND="ssh -i /var/lib/jenkins/.ssh/id_rsa -o StrictHostKeyChecking=no" git clone git@192.168.131.69:xingyun-system/devops.git /tmp/gray_deploy

rm -rf /tmp/gray_deploy /tmp/devops

git clone -b $BRANCH $REPO_URL $WORKDIR

if [ $? -ne 0 ]; then

echo "代码拉取失败"

exit 3

fi

cp -r $WORKDIR /tmp/devops

cd /tmp

tar czf $PACKAGE_NAME devops

rm -rf devops

cd -

}

# 上传包到目标服务器

function upload_package() {

for server in "${SERVERS[@]}"; do

ssh root@$server "mkdir -p $REMOTE_PACKAGES $REMOTE_RELEASES"

scp /tmp/$PACKAGE_NAME root@$server:$REMOTE_PACKAGES/

done

}

# 远程解压并切换软连接，并重启 Tomcat

function remote_deploy_and_link() {

for server in "${SERVERS[@]}"; do

ssh root@$server "

mkdir -p $REMOTE_RELEASES/${DATE_TAG} &&

tar xzf $REMOTE_PACKAGES/$PACKAGE_NAME -C $REMOTE_RELEASES/${DATE_TAG} &&

rm -rf $REMOTE_LINK &&

ln -sfn $REMOTE_RELEASES/${DATE_TAG}/devops $REMOTE_LINK &&

shutdown.sh && startup.sh

"

echo "已部署并切换软连接到 $server，并重启 Tomcat"

done

}

# 回滚到上一个版本（软连接切换到上一个目录/devops，并重启 Tomcat）

function rollback_version() {

for server in "${SERVERS[@]}"; do

PREV=$(ssh root@$server "ls -dt $REMOTE_RELEASES/* | sed -n '2p'")

if [ -z "$PREV" ]; then

echo "$server 没有可回滚的历史版本"

exit 4

fi

ssh root@$server "rm -rf $REMOTE_LINK && ln -sfn $PREV/devops $REMOTE_LINK && shutdown.sh && startup.sh"

echo "$server 已回滚到 $PREV，并重启 Tomcat"

done

}

# 灰度完成后摘除 group1 节点

function offline_group1() {

for name in "${SERVER_NAMES[@]}"; do

ssh root@$HAPROXY_HOST "echo 'disable server $BACKEND_NAME/$name' | socat stdio $HAPROXY_SOCK"

echo "$name 已从 $BACKEND_NAME 摘除"

done

}

# 灰度回滚/上线后重新加入 group1 节点

function online_group1() {

for name in "${SERVER_NAMES[@]}"; do

ssh root@$HAPROXY_HOST "echo 'enable server $BACKEND_NAME/$name' | socat stdio $HAPROXY_SOCK"

echo "$name 已重新加入 $BACKEND_NAME"

done

}

# 主流程

case "$METHOD" in

deploy)

git_and_package

upload_package

remote_deploy_and_link

;;

rollback)

rollback_version

;;

offline)

offline_group1

;;

online)

online_group1

;;

*)

echo "Method参数错误，只能为 deploy、rollback、offline、online"

exit 5

;;

esac

echo "操作完成"

jenkins中构建选择shell执行

bash /var/lib/jenkins/scripts/deploy_gray.sh $METHOD $BRANCH $GROUP

## 4、更新gitlab上的版本

此时tomcat访问的界面如下：

![image.png](media/image7.png)

我们针对这两个问题重新提交一次代码

1、先将项目拉到本地进行修改

git clone http://192.168.131.69/xingyun-system/devops.git # 先将项目拉到本地进行修改

[root@131_70 devops]# cat index.html

<html>

<head>

<meta charset="UTF-8"> # 我已经设置了一个UTF-8了，等下验证下界面是否还会乱码

<title>第一个html程序</title>

</head>

<body>

<h1>hello world!</h1> # 也加了一个标题

hello world! version2 both !!

hello world! version3 both !!

hello world! version4 both !!

</body>

</html>

2、添加至缓存区并提交至本地仓库

git add . && git commit -m "fix title and add h1"

3、提交至远程仓库，即gitlab

git push

注意：一般灰度测试是测试了开发分支develop成功之后才合并至主分支main的,我刚刚一直在主分支下操作

因此我强制更新主分支至开发分支，以便等下用开发分支做测试和部署

[root@131_70 ~]# cd devops/ # 还是当前拉下来的main分支项目

[root@131_70 devops]# git checkout develop # 直接切换至develop分支下

[root@131_70 devops]# git branch # 确认此时在开发分支

* develop

main

[root@131_70 devops]# git reset --hard main # 强制将main分支的内容重置到develop下

[root@131_70 devops]# git push origin develop # 推送至gitlab的develop下

## 9、摘除灰度环境（联动haproxy）

1、在jenkins中增加两个参数选项：offline和online

**offline灰度完成之后，将灰度环境从生产环境中摘除（也就是让vip只负载至68和70，将灰度环境67摘除）**

**online上线时，将灰度环境加入生产的负载中**

![image.png](media/image15.png)

此时我们已经将生产环境升级为最近版本了，但是此时灰度服务器还在负载池中，我们用**offline将灰度环境67从负载池中摘除**

  echo 'disable server group1_backend/tomcat1' | socat stdio
/var/lib/haproxy/haproxy.sock

将灰度环境加入负载池，则用enable

echo 'enable server group1_backend/tomcat1' | socat stdio
/var/lib/haproxy/stats
